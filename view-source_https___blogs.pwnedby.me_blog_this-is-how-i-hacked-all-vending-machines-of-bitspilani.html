<article class="prose col-start-2 max-w-none"> <h2 id="back-in-2024">Back in 2024</h2>
<p>The hack for the vending machine started long back in early 2024 when it was first installed in all hostels. I, along with my wingies, tried some basic price manipulation techniques, though it was not successful at that time because we were rather targeting the big machine instead of understanding the logic behind it and how it works.</p>
<p>The technique which we used was basically scanning the QR code after checking out from the cart in order to pay, and after scanning the QR it generates this address:</p>
<p><strong><code>upi://pay?pa=LENDINGVROTHERSJAIPUR01@ybl&amp;pn=PaymentForLendingVrothersPvt.Ltd&amp;am=40.00&amp;mam=40.00&amp;tr=ST3Y28676390&amp;tn=ST3Y28676390&amp;mc=5046&amp;mode=15&amp;purpose=00</code></strong></p>
<p><img src="/_astro/rkw8qwdpbe.lfAt6CyD_Z1n3gou.webp" alt="image" loading="lazy" decoding="async" width="756" height="1230"></p>
<p>Here I thought that we could probably change the <code>am=40.00</code> and <code>mam=40.00</code> values and that the backend would eventually not verify whether the user had entered a correct amount. This was completely wrong and did not work.</p>
<p>The server was verifying each payment with the <code>transaction_id</code> (<code>tr</code>), which in this case is <code>tr=ST3Y28676390</code>.</p>
<p>If you are curious you can read more about how this <code>transaction_id</code> is generated and what algorithms are used <a href="https://django-payu.readthedocs.io/en/latest/apireference.html" rel="nofollow noreferrer noopener" target="_blank">here</a>.</p>
<p>I had given up testing on the machines as I was struggling to find a way to enter the machine. Also, I did not want to come under everyone’s attention while standing in front of the vendy and trying different things.</p>
<p>Apparently, around mid 2024 some students found a <strong>“hack”</strong> which would basically give them a refund after paying the amount. This hack was basically blocking the sensors which are responsible for checking whether an item was dropped or dispensed, which I think was actually crazy. If I remember correctly, some guy reported this to the company and they patched this method, though I am still not sure.</p>

<h2 id="reconnaissance-20042026">Reconnaissance (20/04/2026)</h2>
<p>All I had was the <code>info@lendingvrothers.com</code> address, which was written on the vending machine, to get started with. As any other bug bounty hunter would, the first step was to gather as many subdomains as possible.</p>
<p><img src="/_astro/HJxf7mdOpZx.B7FWFXcv_2hGQ0E.webp" alt="image" loading="lazy" decoding="async" width="2170" height="725"></p>

<p><strong>What next?</strong></p>
<p>The next possibility I was looking into was figuring out which application or website the vending machine operator uses to fill in the items. <h3 id="discovering-the-real-roadblock-pairip--license-protection">Discovering the Real Roadblock: PairIP / License Protection</h3>
<p>The application was clearly designed to resist modification and sideloaded tampering. As already mentioned above in this blog, I found an interesting code file: <code>com/pairip/licensecheck</code>.</p>
<p>This package contained multiple classes dedicated to licensing, installer verification, and shutdown/paywall handling.</p>
<p>Thanks to this <a href="https://blog.byterialab.com/reversing-googles-new-vm-based-integrity-protection-pairip/" rel="nofollow noreferrer noopener" target="_blank">blog</a> I learned about PairIP protection.</p>
<p>Also this <a href="https://github.com/Solaree/pairipcore" rel="nofollow noreferrer noopener" target="_blank">blog</a>.</p>
<p>Up until this point, I had assumed that once I found the right native library and patched the app, I could simply reinstall it and continue with traffic interception. That assumption turned out to be wrong.</p>

<p>The packaging side was much more annoying than the code patch itself; this was not a normal single-APK Android app. The application was installed as a split package: <code>base.apk</code> and <code>split_config.arm64_v8a.apk</code>.</p>
<p>That meant I could patch the Java/smali logic inside <code>base.apk</code>, but I still had to reinstall the app as a valid split APK set. If either half was wrong, Android would either refuse to install it, or the app would crash before I could continue my analysis.</p>
<p>The app was built with Flutter, so the native side mattered just as much as the Java side. In practice, this meant I had to be extremely careful with the native libraries while rebuilding the APK.</p>
<p><code>libapp.so</code> had to exist at exactly that path inside the APK. Not approximately. Not in some other ABI directory. Not compressed differently. Exactly there. Since I was testing on a Genymotion emulator, everything also had to stay consistent with <code>arm64-v8a</code>. If I broke the ABI layout or repacked the split incorrectly, the app would stop behaving like the original build.</p>

<h2 id="why-caido-was-not-capturing-the-traffic">Why Caido Was Not Capturing the Traffic</h2>
<p>My original plan was to patch the app, run it inside the emulator, point it to Caido, and inspect the traffic. In practice, this turned into one of the most confusing parts of the entire process.</p>
<p>The weird part was that the app was clearly making network requests. I could see it behaving normally, I could trigger flows inside the UI, and <code>adb logcat</code> was also showing activity. But Caido was still not showing the target API traffic.</p>
<p>My first assumption was that if the traffic is not visible in Caido, the app must be rejecting the proxy or rejecting the certificate. The problem was not simply “the app is blocking interception.” The real issue was that the traffic was not reaching Caido in the format Caido expected.</p>
<p><img src="/_astro/rJv-Aju6Zl.Dep2B093_1YVkDs.webp" alt="image" loading="lazy" decoding="async" width="1448" height="1086"></p>
<p>But when traffic is transparently redirected, the client does not know it is talking to a proxy. It thinks it is talking directly to the real server. So instead of sending <code>CONNECT</code>, it sends raw TLS immediately.</p>
<p>That meant Caido was receiving raw TLS on a listener that expected proxy-style traffic.</p>
<p>This was the main reason the setup initially failed. At this point I tried forcing the traffic path externally, but this again caused different problems because Caido expects an explicit proxy handshake and the redirected app traffic was raw TLS, so the two did not match.</p>
<blockquote>
<p>I will test the same workflow with Burp Suite and later add details of it here.</p>
</blockquote>
<p>So to resolve this I wrote a Python script which was acting like a bridge between Caido and Genymotion. The bridge solved the exact protocol mismatch that had been breaking interception before.</p>
<p>Once the interception path finally worked, Caido immediately started showing real application traffic instead of just noise.</p>
